# PORE Methodology

Problem-Oriented Requirements Engineering (PORE) is a methodology that anchors all system requirements in clearly articulated user or stakeholder problems. Its core principle is that every requirement must exist to resolve a specific, stated problem.

## Definition
A **problem** is any observable obstacle, limitation, risk, inefficiency, or unmet need experienced by a user, system, or stakeholder within a given context.

From the Greek problema ("projection"), a problem is not merely a barrier, but a projection that invites resolution. Rooted in classical philosophy, it is a structured challenge that provokes thought, insight, and action. In engineering, a problem defines the space for purposeful design - it is the catalyst for understanding, solving, and building.

### Philosophical Foundation
PORE draws inspiration from Platonic thought, particularly the method of dialectical understanding: first unifying scattered observations into a single idea, then dividing that idea naturally into parts. Similarly, in engineering, we must begin by understanding the problem in full, then structure our solutions along natural boundaries - never forcing divisions that fragment understanding or violate coherence.

## Core Tenets
1. No problem, no requirement - Every requirement must trace back to a problem.
2. Users express problems, not solutions - Stakeholders articulate issues, engineering derives the solutions.
3. Problem statements precede functional analysis - Requirements emerge from the structured understanding of problems.
4. Continuous validation - Problems and their associated requirements must be revisited as context evolves.

## Problem Statement Format
As a [role], I experience [problem], because [underlying reason or constraint].
Examples:
* As a warehouse worker, I can't locate items quickly, because the inventory system lacks real-time location tracking.
* As a product owner, I worry about compliance, because the system doesn't enforce GDPR data handling rules.

## Requirement Derivation
From each problem, derive:
* Root causes
* Success criteria (what does it mean for the problem to be "resolved"?)
* Design space (constraints and freedom for engineering to operate within)
